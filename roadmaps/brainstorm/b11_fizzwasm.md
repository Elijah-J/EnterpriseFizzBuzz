## FizzWASM -- WebAssembly Runtime

### The Problem

The Enterprise FizzBuzz Platform has a bytecode VM (`bytecode_vm.py`) that executes FizzBuzz evaluation programs as custom bytecode instructions. It has a cross-compiler (`cross_compiler.py`) that targets Wasm, C, and Rust output. It has a JIT compiler (`jit_compiler.py`) that generates native machine code at runtime. It has an ELF binary generator (`elf_generator.py`) that produces executable binaries. It has an x86 bootloader (`bootloader.py`) that boots a bare-metal FizzBuzz evaluator. It has a two-pass assembler (`assembler.py`) that assembles x86 machine code. It has a GPU shader compiler (`shader_compiler.py`) that compiles compute shaders. It has a CPU pipeline simulator (`cpu_simulator.py`) that simulates instruction execution through fetch, decode, execute, memory, and writeback stages.

The cross-compiler emits `.wasm` binaries. These binaries have no runtime to execute them. The platform generates WebAssembly bytecode that it cannot run. The `.wasm` output of the cross-compiler is a dead artifact -- a file that exists only to prove the cross-compiler can emit it, not to serve any operational purpose. This is the equivalent of a factory that manufactures car keys but has no cars.

WebAssembly is the universal portable execution format. It runs in browsers, on servers (via WASI runtimes like Wasmtime, Wasmer, WasmEdge), in edge computing nodes, in blockchain smart contracts, in plugin systems, and in embedded devices. The WebAssembly specification defines a stack-based virtual machine with a compact binary format, a well-defined type system, deterministic execution semantics, and a linear memory model with bounds checking. WASI (WebAssembly System Interface) extends WebAssembly with portable system call interfaces: file I/O, environment access, clocks, and process control. The WebAssembly Component Model extends WASI with interface types for language-agnostic module composition.

Real WebAssembly runtimes (Wasmtime, Wasmer, wasm3, WAMR) implement the full WebAssembly 2.0 specification: binary decoding of all 13 section types, validation of type safety and stack consistency, execution of all 400+ instructions, linear memory with page-granularity growth and bounds checking, table-based indirect function calls, multi-value returns, reference types, bulk memory operations, and WASI system call dispatch. The Enterprise FizzBuzz Platform has none of this. It emits `.wasm` files into a void.

### The Vision

A complete WebAssembly runtime implementing the WebAssembly 2.0 binary format specification, a validating decoder for all 13 module sections, a stack-machine interpreter with linear memory, WASI Preview 1 system call support, table-based indirect calls, import/export resolution across module boundaries, fuel-based execution metering for resource limiting, and a Component Model layer for interface type composition. The runtime consumes `.wasm` binaries produced by the cross-compiler and executes them within the platform, closing the loop between compilation and execution. FizzLambda serverless functions gain a WASM execution backend. The capability security system governs WASI capability sets, ensuring that WebAssembly modules can only access the system interfaces explicitly granted to them.

### Key Components

- **`fizzwasm.py`** (~3,500 lines): FizzWASM WebAssembly Runtime

- **WASM Binary Decoder**: Parsing the WebAssembly binary format:
  - **`WasmDecoder`**: a streaming binary decoder that parses `.wasm` files according to the WebAssembly binary format specification. The decoder validates the magic number (`\x00asm`) and version field (`\x01\x00\x00\x00`) at byte offset 0, then reads each section sequentially by section ID and byte length. Unknown section IDs are treated as custom sections and preserved but not interpreted. The decoder produces a `WasmModule` -- an in-memory representation of the decoded module containing all section data in structured form
  - **`WasmModule`**: the root data structure representing a decoded WebAssembly module. Contains: `types` (list of function type signatures), `imports` (list of import descriptors), `functions` (list of function index-to-type-index mappings), `tables` (list of table types), `memories` (list of memory types), `globals` (list of global types with initial values), `exports` (list of export descriptors), `start` (optional start function index), `elements` (list of element segments for table initialization), `code` (list of function bodies with local declarations and instruction sequences), `data` (list of data segments for memory initialization), `custom_sections` (list of named byte blobs)
  - **Section decoders** (one per section type):
    - **Type Section** (ID 1): decodes function type signatures. Each signature is a `FuncType` containing a `params` vector of value types (`i32`, `i64`, `f32`, `f64`, `funcref`, `externref`) and a `results` vector of value types. Multi-value returns are supported (results vector may contain more than one type)
    - **Import Section** (ID 2): decodes import descriptors. Each import specifies a `module` name, a `name` within that module, and a descriptor that is one of: function import (type index), table import (table type), memory import (memory type with initial and optional maximum page count), or global import (value type and mutability)
    - **Function Section** (ID 3): decodes a vector of type indices. Each entry maps a function (by its index in the function index space, offset by the number of imported functions) to its type signature in the type section
    - **Table Section** (ID 4): decodes table type definitions. Each table has an element type (`funcref` or `externref`) and size limits (initial count and optional maximum count). Tables store references for indirect function calls
    - **Memory Section** (ID 5): decodes memory type definitions. Each memory has size limits specified in pages (1 page = 64 KiB). The initial size is the number of pages allocated at instantiation. The optional maximum size caps growth via `memory.grow`
    - **Global Section** (ID 6): decodes global variable definitions. Each global has a value type, a mutability flag (immutable or mutable), and an initializer expression (a constant expression that computes the global's initial value at instantiation time)
    - **Export Section** (ID 7): decodes export descriptors. Each export has a `name` (UTF-8 string) and references one of: a function (by function index), a table (by table index), a memory (by memory index), or a global (by global index). Exports are the module's public interface
    - **Start Section** (ID 8): decodes the optional start function index. If present, this function is invoked automatically after module instantiation, before any exports are called. The start function must have type `[] -> []` (no parameters, no results)
    - **Element Section** (ID 9): decodes element segments that initialize table entries. Each segment specifies a table index, an offset expression (constant expression evaluating to `i32`), and a vector of function indices to write into the table starting at the offset. Active segments are applied during instantiation. Passive segments are applied via `table.init` instructions
    - **Code Section** (ID 10): decodes function bodies. Each body contains a vector of local variable declarations (count and type pairs, compressed) followed by the instruction sequence (expression) that constitutes the function body. Instructions are decoded as opcodes followed by their immediate operands according to the instruction encoding rules
    - **Data Section** (ID 11): decodes data segments that initialize linear memory. Each segment specifies a memory index (typically 0), an offset expression (constant expression evaluating to `i32`), and a byte vector to write into memory at the offset. Active segments are applied during instantiation. Passive segments are applied via `memory.init` instructions
    - **Custom Section** (ID 0): decodes named custom sections. The section payload is a UTF-8 name string followed by an opaque byte sequence. Custom sections carry metadata (e.g., `name` section for debug symbols, `producers` section for toolchain info) but do not affect module semantics. The decoder preserves all custom sections in the `WasmModule` for diagnostic and debugging purposes

- **Module Validation**: Type checking and structural verification:
  - **`WasmValidator`**: validates a decoded `WasmModule` for structural and type correctness before execution. Validation rejects modules that would cause undefined behavior during execution. A module that passes validation is guaranteed to execute without type errors, stack underflows, or out-of-bounds accesses to the operand stack (memory bounds checking is still performed at runtime because memory addresses are computed dynamically)
  - **Type validation**: verifies that all type indices referenced by functions, imports, and call instructions are within the bounds of the type section. Verifies that function signatures in indirect call sites (`call_indirect`) match the expected type at runtime dispatch (encoded as an immediate type index in the instruction)
  - **Stack machine verification**: the core validation algorithm. Implements the WebAssembly type-checking algorithm described in the specification's appendix. Processes each instruction in each function body, maintaining an abstract operand stack of value types. Each instruction pops its expected input types from the abstract stack and pushes its result types. The validator rejects functions where:
    - An instruction pops from an empty stack (stack underflow)
    - The types on the stack do not match the instruction's expected input types (type mismatch)
    - At the end of a block/function, the stack does not match the block/function's result type signature
    - After a branch instruction, unreachable code contains instructions that would produce inconsistent stack states across different control flow paths
  - **Control flow validation**: verifies that structured control flow instructions (`block`, `loop`, `if`/`else`/`end`, `br`, `br_if`, `br_table`) form well-nested structures. Branch targets are validated to reference enclosing blocks. Branch target label indices are checked against the control frame stack depth. `br_table` label vectors are validated to ensure all targets have compatible stack types
  - **Import/export validation**: verifies that import descriptors reference valid external types and that export names are unique within the module. Verifies that the start function, if present, has the correct type signature `[] -> []`
  - **Memory and table limits validation**: verifies that initial sizes do not exceed maximum sizes, that maximum sizes do not exceed implementation limits (65,536 pages for memory = 4 GiB, configurable for tables), and that at most one memory is defined (WebAssembly 2.0 multi-memory proposal is not yet standardized)
  - **Global initialization validation**: verifies that global initializer expressions contain only constant instructions (`i32.const`, `i64.const`, `f32.const`, `f64.const`, `global.get` of an imported immutable global, `ref.null`, `ref.func`). Mutable imported globals referenced in initializers are rejected. This ensures that global initialization is deterministic and does not depend on execution state
  - **Data and element segment validation**: verifies that segment offsets are constant expressions evaluating to valid memory/table indices, that data segment byte vectors do not exceed memory bounds (checked at instantiation time for active segments), and that element segment function indices are valid

- **Stack Machine Interpreter**: WebAssembly instruction execution:
  - **`WasmInterpreter`**: a stack-based interpreter that executes validated WebAssembly modules. The interpreter maintains an operand stack (a vector of `WasmValue` -- tagged union of `i32`, `i64`, `f32`, `f64`, `funcref`, `externref`), a call stack (vector of `CallFrame` -- function index, local variables, return address, operand stack base pointer), and a control stack (vector of `ControlFrame` -- block kind, label target, result arity, operand stack height at block entry)
  - **Instruction dispatch**: the interpreter decodes and dispatches instructions from the function body's bytecode. Instruction categories:
    - **Numeric instructions**: `i32.add`, `i32.sub`, `i32.mul`, `i32.div_s`, `i32.div_u`, `i32.rem_s`, `i32.rem_u`, `i32.and`, `i32.or`, `i32.xor`, `i32.shl`, `i32.shr_s`, `i32.shr_u`, `i32.rotl`, `i32.rotr`, `i32.clz`, `i32.ctz`, `i32.popcnt`, `i32.eqz`, `i32.eq`, `i32.ne`, `i32.lt_s`, `i32.lt_u`, `i32.gt_s`, `i32.gt_u`, `i32.le_s`, `i32.le_u`, `i32.ge_s`, `i32.ge_u`, `i32.wrap_i64`, `i32.trunc_f32_s`, `i32.trunc_f64_s`, `i32.extend8_s`, `i32.extend16_s`, `i32.reinterpret_f32`. Equivalent 64-bit variants for `i64`. Floating-point variants for `f32`/`f64` including `abs`, `neg`, `ceil`, `floor`, `trunc`, `nearest`, `sqrt`, `min`, `max`, `copysign`. All arithmetic follows WebAssembly semantics: integer division by zero traps, integer overflow wraps, floating-point operations follow IEEE 754-2019
    - **Memory instructions**: `i32.load`, `i64.load`, `f32.load`, `f64.load`, `i32.store`, `i64.store`, `f32.store`, `f64.store`, and their 8/16/32-bit load/store variants with sign extension. Each instruction has an alignment hint (a power-of-two byte alignment, advisory only) and an offset (added to the dynamic address on the stack). Effective address = `stack_address + offset`. Bounds checking: if `effective_address + access_size > memory_size`, the instruction traps
    - **Control flow instructions**: `block`, `loop`, `if`, `else`, `end`, `br`, `br_if`, `br_table`, `return`, `call`, `call_indirect`, `unreachable`, `nop`. The interpreter manages control frames: `block` pushes a frame targeting the end of the block; `loop` pushes a frame targeting the start of the loop; `if` pushes a conditional frame. `br N` unwinds N+1 control frames, restoring the operand stack to the target frame's height (preserving result values). `call` pushes a new call frame with fresh locals. `call_indirect` reads a function index from the table, checks the function's type against the expected type index, and calls the function. Type mismatch traps
    - **Variable instructions**: `local.get`, `local.set`, `local.tee`, `global.get`, `global.set`. Locals are indexed per call frame. Globals are module-wide. Attempting to set an immutable global traps
    - **Reference instructions**: `ref.null`, `ref.is_null`, `ref.func`. Create null references, test for null, and create function references
    - **Table instructions**: `table.get`, `table.set`, `table.size`, `table.grow`, `table.fill`, `table.copy`, `table.init`, `elem.drop`. Table operations for managing indirect function call tables
    - **Memory management instructions**: `memory.size` (returns current memory size in pages), `memory.grow` (attempts to grow memory by N pages, returns previous size or -1 on failure), `memory.fill`, `memory.copy`, `memory.init`, `data.drop`. Bulk memory operations for efficient data movement
  - **Trap handling**: certain conditions cause the interpreter to trap (abort execution with an error): integer division by zero, integer overflow in `trunc` conversions, out-of-bounds memory access, out-of-bounds table access, `call_indirect` type mismatch, stack overflow (call depth exceeds limit), unreachable instruction reached, fuel exhaustion. Traps are propagated as `WasmTrap` exceptions containing the trap reason, the instruction offset, and the call stack at the point of the trap

- **Linear Memory Model**: Bounds-checked byte-addressable memory:
  - **`LinearMemory`**: represents a WebAssembly linear memory instance. Memory is a contiguous byte array organized into 64 KiB pages. The memory is initialized with the number of pages specified by the memory type's initial size. Data segments are copied into memory during instantiation
  - **Bounds checking**: every memory access (load, store, memory.copy, memory.fill, memory.init) is bounds-checked. The effective address plus the access size must not exceed the current memory size in bytes. Out-of-bounds access traps immediately. There is no undefined behavior -- the bounds check is mandatory, not optional
  - **`memory.grow`**: attempts to grow the memory by a specified number of pages. If the growth would exceed the memory type's maximum page count (or the implementation limit of 65,536 pages = 4 GiB), the operation fails and returns -1 (as an `i32` on the operand stack). On success, the new pages are zero-initialized and the previous page count is returned. Growth is atomic: it either succeeds completely or fails completely with no partial allocation
  - **Memory isolation**: each module instance has its own linear memory. Modules cannot access each other's memory directly. Inter-module data exchange occurs exclusively through function calls (passing values on the operand stack) or through shared memory (if explicitly configured via import/export). This isolation is foundational to WebAssembly's security model

- **Table and Indirect Calls**: Function reference management:
  - **`WasmTable`**: represents a WebAssembly table instance. Tables are arrays of typed references (`funcref` or `externref`). Tables enable indirect function calls: the `call_indirect` instruction reads a function reference from the table at a dynamic index, checks its type against the expected type, and invokes the referenced function
  - **Element segment initialization**: during module instantiation, element segments write function references into tables at specified offsets. This is analogous to how data segments write bytes into memory. Active element segments are applied automatically; passive segments are applied via `table.init`
  - **`call_indirect` dispatch**: the instruction pops a table index from the operand stack, reads the function reference at that index from the specified table, verifies that the referenced function's type matches the expected `FuncType` (specified by the instruction's immediate type index), and invokes the function. If the table entry is null (`ref.null`) or the type does not match, the instruction traps. This type-checked indirect call mechanism supports safe dynamic dispatch (virtual method tables, function pointers, callbacks) without sacrificing type safety
  - **Table growth**: `table.grow` extends the table by N entries, initialized with a specified reference value (typically `ref.null`). Returns the previous table size or -1 on failure. Growth semantics mirror `memory.grow`

- **Import/Export Resolution**: Cross-module linking:
  - **`ImportResolver`**: resolves a module's import declarations against a provided set of host functions and other module instances. Each import descriptor (`module_name`, `import_name`, `descriptor_kind`) is matched against the import environment:
    - **Function imports**: matched by module name and function name. The resolved function must have a type signature that matches the import's declared type index. Host functions (Python callables wrapped in `HostFunction`) are type-checked against the expected signature at link time
    - **Memory imports**: matched by name. The provided memory must have limits that are at least as large as the import's declared limits (initial >= import initial, maximum <= import maximum or both absent)
    - **Table imports**: matched by name. The provided table must have compatible element type and limits
    - **Global imports**: matched by name. The provided global must have matching value type and mutability
  - **`ExportSet`**: the set of values exported by an instantiated module. Exports are the public interface through which other modules (or host code) interact with the instance. Export values include: exported functions (callable with type checking), exported memories (accessible for host-side reads/writes), exported tables, and exported globals
  - **Module linking**: multiple modules can be linked by providing one module's exports as another module's imports. The `ImportResolver` verifies type compatibility at link time, before any code executes. This enables modular WebAssembly application composition where subsystems are compiled as separate `.wasm` modules and linked at load time

- **Module Instantiation**: Turning a decoded module into an executable instance:
  - **`ModuleInstance`**: the runtime representation of an instantiated WebAssembly module. Contains: resolved imports, function instances (code bodies bound to their module instance), memory instances (allocated and initialized linear memories), table instances (allocated and initialized with element segments), global instances (allocated and initialized with evaluated constant expressions), and the export set
  - **Instantiation sequence**:
    1. **Validate** the module (if not already validated)
    2. **Resolve imports** against the provided import environment, verifying type compatibility for all imports
    3. **Allocate memories** with the declared initial page counts, zero-initialized
    4. **Allocate tables** with the declared initial element counts, null-initialized
    5. **Allocate globals** and evaluate initializer expressions to compute initial values (only constants and imported global reads are permitted in initializers, ensuring determinism)
    6. **Apply active data segments** -- copy byte vectors into memories at their offset expressions. Trap if any segment's offset + length exceeds the memory size
    7. **Apply active element segments** -- write function references into tables at their offset expressions. Trap if any segment's offset + count exceeds the table size
    8. **Build the export set** from the module's export declarations, binding each export name to the corresponding function/memory/table/global instance
    9. **Invoke the start function**, if present. The start function runs with no arguments and must return no values. If the start function traps, instantiation fails
  - **Instance isolation**: each call to `instantiate` produces a fresh `ModuleInstance` with its own memories, tables, and mutable globals. Two instances of the same module do not share state (unless a memory or table is explicitly shared via imports). This enables safe concurrent instantiation of the same module for different FizzBuzz evaluation contexts

- **WASI Preview 1**: Portable system interface:
  - **`WasiPreview1`**: implements WASI Preview 1 (snapshot `wasi_snapshot_preview1`) system calls as host-imported functions. WASI provides a capability-based system interface: each system call operates on resources (file descriptors, directories, clocks) that must be explicitly provided to the module. The WASI implementation is the bridge between the sandboxed WebAssembly execution environment and the host platform's resources
  - **`fd_read`** (`fd: i32, iovs: i32, iovs_len: i32, nread: i32 -> errno: i32`): reads data from a file descriptor into a scatter/gather vector of memory buffers. The `iovs` parameter points to an array of `(buf_ptr: i32, buf_len: i32)` pairs in linear memory. Each pair describes a buffer into which data should be read. The function reads sequentially into the buffers and writes the total bytes read to `nread`. File descriptors 0 (stdin), 1 (stdout), and 2 (stderr) are pre-opened. Returns `errno` 0 on success, 8 (`BADF`) for invalid file descriptors, 28 (`INVAL`) for invalid arguments
  - **`fd_write`** (`fd: i32, iovs: i32, iovs_len: i32, nwritten: i32 -> errno: i32`): writes data from a gather vector of memory buffers to a file descriptor. The gather vector format matches `fd_read`. This is the primary output mechanism for WASM modules: writing FizzBuzz evaluation results to stdout (fd 1). Writes to stdout are captured by the runtime and routed to the platform's output pipeline
  - **`args_get`** (`argv: i32, argv_buf: i32 -> errno: i32`): retrieves command-line arguments. Writes an array of pointers into `argv` and the argument strings (null-terminated) into `argv_buf`. The module receives the arguments that were configured when the WASM instance was created, enabling parameterized FizzBuzz evaluation (e.g., range start, range end, output format)
  - **`args_sizes_get`** (`argc: i32, argv_buf_size: i32 -> errno: i32`): retrieves the number of arguments and the total byte size of argument strings. The module calls this before `args_get` to allocate sufficient buffer space
  - **`environ_get`** (`environ: i32, environ_buf: i32 -> errno: i32`): retrieves environment variables. Format: array of `KEY=VALUE\0` strings. Environment variables are filtered by the capability set -- only explicitly granted environment variables are visible to the module
  - **`environ_sizes_get`** (`environc: i32, environ_buf_size: i32 -> errno: i32`): retrieves the count and total byte size of environment variables
  - **`clock_time_get`** (`clock_id: i32, precision: i64, timestamp: i32 -> errno: i32`): retrieves the current time from a clock. Supported clock IDs: `CLOCK_REALTIME` (wall clock), `CLOCK_MONOTONIC` (monotonic timer), `CLOCK_PROCESS_CPUTIME_ID` (process CPU time), `CLOCK_THREAD_CPUTIME_ID` (thread CPU time). Time is returned in nanoseconds. The monotonic clock is used for performance measurement within WASM modules. The realtime clock is used for timestamped FizzBuzz evaluation output
  - **`proc_exit`** (`exitcode: i32 -> noreturn`): terminates the WASM instance with the given exit code. Raises a `WasmProcExit` exception that is caught by the runtime to cleanly terminate execution. Exit code 0 indicates successful FizzBuzz evaluation; non-zero indicates failure
  - **`random_get`** (`buf: i32, buf_len: i32 -> errno: i32`): writes cryptographically secure random bytes into the specified memory buffer. Used by modules that implement randomized FizzBuzz evaluation strategies (e.g., genetic algorithm fitness evaluation, chaos engineering entropy source)
  - **WASI capability enforcement**: the `WasiPreview1` implementation receives a `WasiCapabilities` configuration at instantiation that specifies which capabilities the module is granted: `allow_fd_read` (list of allowed file descriptors for reading), `allow_fd_write` (list of allowed file descriptors for writing), `allow_env` (list of allowed environment variable names), `allow_clocks` (list of allowed clock IDs), `allow_random` (whether `random_get` is permitted). System calls that access ungrated capabilities return `errno` 76 (`NOTCAPABLE`). This integrates with the platform's capability security system (FizzCap): WASI capabilities are derived from the module's capability token

- **Fuel-Based Execution Metering**: Resource limiting for untrusted code:
  - **`FuelMeter`**: tracks and limits the computational resources consumed by a WASM execution. Each instruction execution consumes one unit of fuel. When fuel reaches zero, the interpreter traps with `FuelExhausted`. This prevents infinite loops, excessive computation, and denial-of-service from malicious or buggy WASM modules
  - **Fuel allocation**: each `ModuleInstance` is created with a configurable fuel budget (default: 10,000,000 instructions -- sufficient for evaluating FizzBuzz across a large range). Fuel can be replenished between invocations. The fuel budget is a hard limit: exceeding it is a trap, not a warning
  - **Fuel cost model**: basic instructions (arithmetic, comparisons, local/global access) cost 1 fuel. Memory access instructions cost 2 fuel (reflecting the bounds check overhead). Function calls cost 5 fuel (reflecting the call frame allocation and type checking). Host function calls (WASI calls) cost 10 fuel (reflecting the sandbox boundary crossing). The cost model is configurable, enabling fine-grained resource accounting for different execution contexts
  - **Fuel checkpoints**: the interpreter checks fuel at configurable intervals (default: every instruction). For performance-sensitive execution, the check interval can be increased (e.g., every 1000 instructions), trading granularity for throughput. Checkpoint-based metering ensures that fuel exhaustion is detected within `check_interval` instructions of the limit
  - **Integration with FizzLambda**: FizzLambda serverless functions that use the WASM execution backend receive their fuel budget from FizzLambda's resource quota system. This creates a unified resource model: FizzLambda manages execution time, and FizzWASM enforces it at the instruction level via fuel metering

- **WASM Component Model**: Interface types for module composition:
  - **`ComponentModel`**: implements a subset of the WebAssembly Component Model specification, enabling high-level module composition with interface types. The Component Model sits above the core WebAssembly layer and provides a language-agnostic way to define, implement, and consume interfaces between modules
  - **`InterfaceType`**: extends the core WebAssembly value types (`i32`, `i64`, `f32`, `f64`, `funcref`, `externref`) with high-level types: `string` (UTF-8 encoded), `list<T>` (variable-length sequence), `record` (named fields), `variant` (tagged union), `enum` (named alternatives), `option<T>` (nullable), `result<T, E>` (success or error), `tuple` (anonymous product). Interface types are lowered to core WebAssembly types at the component boundary: strings become `(i32, i32)` pointer/length pairs pointing into linear memory, lists become `(i32, i32)` pointer/count pairs, records become flattened sequences of core types
  - **`WIT` (WebAssembly Interface Type) parser**: parses WIT definitions that describe component interfaces. A WIT definition specifies: the interface name, the types used in the interface, the functions exposed by the interface (with parameter and return types expressed in interface types), and the imports and exports of the component. Example WIT for a FizzBuzz evaluator component:
    ```
    interface fizzbuzz-evaluator {
      record evaluation-result {
        number: u32,
        output: string,
        classification: string,
      }
      evaluate: func(n: u32) -> evaluation-result
      evaluate-range: func(start: u32, end: u32) -> list<evaluation-result>
    }
    ```
  - **Component instantiation**: a component is instantiated by linking its WIT-declared imports to implementations (other components or host functions). The Component Model layer handles the lifting (core types to interface types) and lowering (interface types to core types) at each component boundary. String arguments are copied into the callee's linear memory. List results are copied out of the callee's linear memory. This memory isolation ensures that components cannot access each other's internal state through shared memory
  - **Canonical ABI**: the deterministic mapping between interface types and core types, as specified by the Component Model canonical ABI. The ABI defines: how strings are passed (UTF-8 bytes, pointer + length, callee allocates via `canonical_abi_realloc`), how lists are passed (element sequence, pointer + count), how records are flattened (fields in declaration order), how variants are represented (discriminant + payload), and how results are returned (0 for success + value, 1 for error + error value). The runtime implements both `canon lift` (core to interface) and `canon lower` (interface to core) operations

- **Integration Points**:
  - **Cross-compiler integration**: the cross-compiler's `.wasm` output target produces valid WebAssembly modules that FizzWASM can decode, validate, and execute. The integration test compiles a FizzBuzz evaluator to `.wasm` via the cross-compiler and executes it via FizzWASM, verifying end-to-end correctness. The cross-compiler emits valid WASI imports for system calls used by the compiled program
  - **FizzLambda WASM backend**: FizzLambda serverless functions gain a `runtime: wasm` configuration option. When set, FizzLambda compiles the function to `.wasm` via the cross-compiler, instantiates it via FizzWASM with WASI capabilities derived from the function's permission set, and executes invocations by calling the module's exported entry point. Each invocation gets a fresh module instance (memory isolation) with a fuel budget derived from the function's timeout configuration
  - **Capability security integration**: WASI capabilities are governed by FizzCap capability tokens. When a WASM module is instantiated, the runtime requests a capability token from FizzCap specifying the required WASI capabilities (file descriptor access, environment variable access, clock access, random access). The capability token is validated against the module's permission policy. Revoked capabilities take effect at the next WASI call (the `WasiPreview1` implementation checks the token's validity on each call)

- **FizzWASM Middleware**: `FizzWASMMiddleware` integrates with the middleware pipeline, annotating FizzBuzz evaluation responses with WASM execution metadata: module name, fuel consumed, fuel remaining, peak memory pages, and execution time in microseconds. When the WASM execution backend is active, the middleware records per-evaluation WASM metrics for performance analysis

- **CLI Flags**: `--fizzwasm`, `--fizzwasm-run <file.wasm>` (decode, validate, and execute a WASM module with default WASI capabilities), `--fizzwasm-validate <file.wasm>` (validate without executing), `--fizzwasm-inspect <file.wasm>` (display module sections: types, imports, exports, memory limits, table definitions, custom sections), `--fizzwasm-fuel <N>` (set fuel budget for execution, default 10000000), `--fizzwasm-wasi-stdin <file>` (redirect WASI stdin from file), `--fizzwasm-wasi-env <KEY=VALUE>` (add environment variable to WASI), `--fizzwasm-wasi-args <args>` (set command-line arguments for WASI `args_get`), `--fizzwasm-compile-and-run` (compile current FizzBuzz configuration to WASM via cross-compiler and execute via FizzWASM), `--fizzwasm-component <wit_file>` (load a WIT interface definition for component model linking), `--fizzwasm-no-validate` (skip validation for pre-validated modules, performance optimization), `--fizzwasm-fuel-cost-model <model>` (select fuel cost model: `uniform`, `weighted`, or `custom`)

### Why This Is Necessary

Because the platform compiles to WebAssembly but cannot run WebAssembly. The cross-compiler emits `.wasm` binaries. The JIT compiler generates native code. The bytecode VM executes custom bytecode. The ELF generator produces Linux executables. The assembler emits x86 machine code. The bootloader boots on bare metal. The GPU shader compiler targets compute shaders. Every compilation target has a corresponding execution environment -- except WebAssembly. The cross-compiler's WASM backend produces files that leave the platform and enter a void. There is no runtime to load them, no validator to check them, no interpreter to execute them.

WebAssembly is not an incidental compilation target. It is the only bytecode format with universal adoption across browsers, servers, edge nodes, blockchains, and embedded devices. The platform's commitment to computational universality -- evidenced by its bytecode VM, JIT compiler, ELF generator, assembler, CPU simulator, and bootloader -- is incomplete without a WASM runtime. The capability-based security model of WASI aligns directly with FizzCap's capability security system. The Component Model's interface types enable modular FizzBuzz evaluation across independently compiled WASM components. The fuel-based metering provides deterministic resource limiting for FizzLambda serverless functions. FizzWASM closes the compilation-execution loop and makes the platform a complete WebAssembly host environment.

### Estimated Scale

~3,500 lines of WebAssembly runtime, ~450 lines of binary decoder (magic number validation, section parsing for all 13 section types, LEB128 encoding, instruction decoding), ~350 lines of module validation (type checking, stack machine verification, control flow nesting, import/export validation, limit checking, constant expression validation), ~700 lines of stack machine interpreter (instruction dispatch for numeric, memory, control flow, variable, reference, table, and bulk memory instructions, operand stack management, call frames, control frames, trap handling), ~250 lines of linear memory model (page-based allocation, bounds checking, grow semantics, data segment initialization), ~200 lines of table and indirect calls (funcref tables, element segment initialization, call_indirect dispatch, table growth), ~200 lines of import/export resolution (type-compatible matching, host function wrapping, multi-module linking), ~150 lines of module instantiation (allocation sequence, segment application, start function invocation, instance isolation), ~350 lines of WASI Preview 1 (fd_read, fd_write, args_get, args_sizes_get, environ_get, environ_sizes_get, clock_time_get, proc_exit, random_get, capability enforcement), ~150 lines of fuel-based metering (fuel counter, cost model, checkpoint checking, FizzLambda integration), ~300 lines of Component Model (InterfaceType definitions, WIT parser, canonical ABI lift/lower, component instantiation), ~150 lines of middleware and CLI integration, ~500 tests. Total: ~4,750 lines.
