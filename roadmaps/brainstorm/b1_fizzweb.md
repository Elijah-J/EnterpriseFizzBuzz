## Idea: FizzWeb -- Production HTTP/HTTPS Web Server

### The Problem

The Enterprise FizzBuzz Platform has a reverse proxy (FizzProxy) that distributes evaluation requests across backend rule engine instances. It has an API gateway that normalizes, enriches, validates, and transforms requests. It has a TCP/IP stack (FizzNet) that implements reliable delivery from Ethernet frames through TCP segments. It has an HTTP/2 protocol module referenced in the compose service catalog. It has a DNS server (FizzDNS) that resolves hostnames. It has a container networking interface (FizzCNI) that connects containers via bridge, overlay, and host network drivers.

It has no HTTP server.

The platform can route HTTP requests to backends that do not exist. It can load-balance traffic across pools that cannot accept connections. It can resolve domain names to IP addresses that no server is listening on. It can configure container network interfaces with port mappings that map to nothing. The API gateway transforms requests that no server receives. The reverse proxy health-checks backends that have no listening socket. The service mesh establishes sidecar proxies alongside services that cannot serve HTTP.

This is the architectural equivalent of building an airport terminal with gates, runways, air traffic control, baggage handling, customs, and ticketing -- but no aircraft. Every supporting system is operational. The core transport mechanism is absent.

Real web infrastructure begins with an HTTP server. Apache HTTP Server, NGINX, Caddy, Lighttpd -- these are the foundational components upon which reverse proxies, API gateways, load balancers, and service meshes are built. The HTTP server is the process that binds to a port, accepts TCP connections, parses HTTP request messages, routes them to handlers, generates HTTP response messages, and sends them back over the connection. Every other component in the request pipeline -- TLS termination, virtual host routing, compression, rate limiting, access logging, keep-alive management -- is a feature of the HTTP server or a middleware layered on top of it. Without an HTTP server, the platform's entire network infrastructure operates in a vacuum: FizzProxy routes requests to backends that have no HTTP listener, FizzNet delivers TCP segments to ports that no process has bound, and FizzCNI maps container ports to host ports where no server accepts connections.

The platform processes 300,000+ lines of code to evaluate whether numbers are divisible by 3 or 5. These evaluation results must be served over HTTP to external clients. The FizzBuzz-as-a-Service vision from Round 1 established the API surface. The API gateway from the infrastructure layer defined the request/response transformation pipeline. But the actual HTTP server -- the process that opens a socket, performs the TLS handshake, parses `GET /api/v1/evaluate?n=15 HTTP/1.1`, dispatches to the evaluation engine, and responds with `{"result": "FizzBuzz"}` -- does not exist.

### The Vision

A production-grade HTTP/1.1 and HTTP/2 web server for the Enterprise FizzBuzz Platform, implementing the full HTTP specification with TLS termination, virtual host routing, static file serving, CGI/WSGI interface, WebSocket upgrade, connection pooling, keep-alive management, chunked transfer encoding, gzip/deflate/brotli compression, structured access logging, rate limiting integration, middleware pipeline, and graceful shutdown. FizzWeb is the platform's external-facing network endpoint -- the process that binds to port 8080 (HTTP) and port 8443 (HTTPS), accepts TCP connections via FizzNet, terminates TLS for encrypted connections, parses HTTP request messages according to RFC 7230-7235 and RFC 9113, routes requests to handlers based on virtual host configuration and URL path matching, invokes the FizzBuzz evaluation engine for API requests, serves static assets (the platform dashboard, documentation, and FizzSheet exports) for file requests, and returns well-formed HTTP response messages with appropriate status codes, headers, and entity bodies.

The server integrates with every relevant platform subsystem: FizzNet for TCP transport, FizzCNI for container port binding, FizzProxy for upstream load balancing, the API gateway for request transformation, the rate limiter for traffic shaping, FizzOTel for distributed tracing, the access log feeds into the event sourcing journal, TLS certificates are retrieved from the secrets vault, virtual host configurations are managed through the configuration manager, and graceful shutdown coordinates with FizzContainerd's container lifecycle signals. FizzWeb is the missing aircraft that makes the airport operational.

### Key Components

- **`fizzweb.py`** (~3,500 lines): FizzWeb Production HTTP/HTTPS Web Server
- **HTTP Request Parser**: RFC 7230-compliant HTTP/1.1 message parser:
  - **`HTTPRequestParser`**: parses raw byte streams from FizzNet TCP connections into structured `HTTPRequest` objects. Implements the HTTP/1.1 message format: request-line (`method SP request-target SP HTTP-version CRLF`), header fields (`field-name ":" OWS field-value OWS CRLF`), and optional message body. Handles all standard methods: `GET`, `HEAD`, `POST`, `PUT`, `DELETE`, `PATCH`, `OPTIONS`, `TRACE`, `CONNECT`. Validates request-line syntax, enforces maximum header size (8KB default, configurable), maximum request body size (10MB default, configurable), and maximum header count (100 default). Rejects malformed requests with `400 Bad Request`. Supports absolute-form and origin-form request targets. Parses `Transfer-Encoding: chunked` bodies by reading chunk-size, chunk-data, and trailer fields per RFC 7230 Section 4.1. Detects and rejects request smuggling attempts (conflicting `Content-Length` and `Transfer-Encoding` headers)
  - **`HTTPRequest`**: data model representing a parsed HTTP request: method, path, query parameters (parsed from the query string), HTTP version, headers (case-insensitive multi-map), body (bytes), remote address, and timestamp. Provides accessor methods for common headers: `Host`, `Content-Type`, `Content-Length`, `Accept`, `Accept-Encoding`, `Authorization`, `Cookie`, `User-Agent`, `Connection`, `Upgrade`
  - **`HTTP2RequestParser`**: parses HTTP/2 frames into `HTTPRequest` objects. Leverages the platform's existing HTTP/2 frame types (`HEADERS`, `DATA`, `SETTINGS`, `WINDOW_UPDATE`, `PRIORITY`, `RST_STREAM`, `GOAWAY`, `PING`) for multiplexed request handling. Performs HPACK header decompression, stream dependency resolution, and flow control window management. Converts HTTP/2 pseudo-headers (`:method`, `:path`, `:scheme`, `:authority`) into the standard `HTTPRequest` model for uniform handler dispatch
- **HTTP Response Builder**: Structured HTTP response construction:
  - **`HTTPResponse`**: data model representing an HTTP response: status code, reason phrase, headers (multi-map), body (bytes or streaming iterator), and HTTP version. Factory methods for common responses: `ok(body)`, `created(location)`, `no_content()`, `redirect(location, permanent=False)`, `bad_request(message)`, `unauthorized(realm)`, `forbidden()`, `not_found()`, `method_not_allowed(allowed)`, `internal_server_error()`, `service_unavailable(retry_after)`
  - **`HTTPResponseSerializer`**: serializes `HTTPResponse` objects into byte streams for transmission over FizzNet TCP connections. Generates status-line, serializes headers, applies `Transfer-Encoding: chunked` for streaming bodies of unknown length, sets `Content-Length` for fixed-size bodies. For HTTP/2 connections, serializes responses into HEADERS and DATA frames with HPACK header compression and flow control compliance
- **TLS Termination**: SSL/TLS handshake and encrypted transport:
  - **`TLSTerminator`**: implements TLS 1.2 and TLS 1.3 handshake simulation for HTTPS connections on port 8443. Retrieves server certificates and private keys from the platform's secrets vault (`FizzVaultManager`). Supports Server Name Indication (SNI) for virtual host certificate selection -- the `ClientHello` message's SNI extension determines which certificate is presented, enabling multiple HTTPS virtual hosts on a single IP address. Cipher suite negotiation selects from a hardened list: `TLS_AES_256_GCM_SHA384`, `TLS_CHACHA20_POLY1305_SHA256`, `TLS_AES_128_GCM_SHA256` for TLS 1.3; `ECDHE-RSA-AES256-GCM-SHA384`, `ECDHE-RSA-AES128-GCM-SHA256` for TLS 1.2. Enforces HSTS (`Strict-Transport-Security: max-age=31536000; includeSubDomains`) on all HTTPS responses. Redirects HTTP requests to HTTPS when `--fizzweb-force-tls` is enabled
  - **`CertificateManager`**: manages TLS certificate lifecycle: generation of self-signed certificates for development, loading of CA-signed certificates from the secrets vault for production, automatic certificate rotation when certificates approach expiration (30 days before expiry by default), and OCSP stapling simulation for certificate revocation checking
- **Virtual Host Routing**: Name-based and IP-based virtual hosts:
  - **`VirtualHost`**: configuration for a single virtual host: `server_name` (hostname pattern, supports wildcards like `*.fizzbuzz.enterprise`), `document_root` (path for static file serving), `routes` (URL pattern to handler mappings), `tls_certificate` (certificate reference for HTTPS), `access_log` (per-host access log configuration), `error_pages` (custom error page paths for specific status codes), `rate_limit_profile` (per-host rate limiting configuration)
  - **`VirtualHostRouter`**: resolves incoming requests to virtual hosts based on the `Host` header (HTTP/1.1) or `:authority` pseudo-header (HTTP/2). Supports exact match (`api.fizzbuzz.enterprise`), wildcard match (`*.fizzbuzz.enterprise`), and a default virtual host for requests that match no configured host. Returns `421 Misdirected Request` if TLS SNI hostname does not match the `Host` header (RFC 7540 Section 9.1.2)
  - **Default virtual hosts**: four pre-configured hosts: `api.fizzbuzz.enterprise` (API endpoint for FizzBuzz evaluation, routes to the evaluation engine), `dashboard.fizzbuzz.enterprise` (serves the platform monitoring dashboard), `docs.fizzbuzz.enterprise` (serves platform documentation and FizzSheet exports), `default` (catch-all host returning the platform status page)
- **Static File Serving**: Efficient file response generation:
  - **`StaticFileHandler`**: serves files from the virtual host's `document_root` directory. Resolves URL paths to filesystem paths with directory traversal protection (rejects paths containing `..` or null bytes). Returns appropriate `Content-Type` headers based on file extension (`text/html`, `text/css`, `application/javascript`, `application/json`, `image/png`, `image/svg+xml`, `application/pdf`, etc.). Supports `If-Modified-Since` conditional requests (returns `304 Not Modified` when the file has not changed). Supports `Range` requests for partial content delivery (returns `206 Partial Content` with the requested byte range). Generates directory index pages when `--fizzweb-autoindex` is enabled. Sets `Cache-Control` headers based on file type: immutable assets (hashed filenames) get `max-age=31536000`, HTML gets `no-cache`, API responses get `no-store`
  - **`MIMETypeRegistry`**: maps file extensions to MIME types. Contains 47 standard mappings plus platform-specific types: `.fizztranslation` maps to `application/x-fizzbuzz-locale`, `.fizzfile` maps to `application/x-fizzfile-build`, `.fizzbuzz` maps to `application/x-fizzbuzz-result`
- **CGI/WSGI Interface**: Application server integration:
  - **`WSGIAdapter`**: implements the WSGI (PEP 3333) interface, allowing the FizzBuzz evaluation engine and any WSGI-compliant application to be served by FizzWeb. Constructs the WSGI `environ` dictionary from the `HTTPRequest` (including `REQUEST_METHOD`, `SCRIPT_NAME`, `PATH_INFO`, `QUERY_STRING`, `SERVER_NAME`, `SERVER_PORT`, `HTTP_*` headers, `wsgi.input`, `wsgi.errors`, `wsgi.url_scheme`). Invokes the WSGI application callable and converts the response iterator into an `HTTPResponse`. Handles `Content-Length` vs. chunked transfer based on whether the application provides the header
  - **`CGIHandler`**: implements CGI (RFC 3875) for executing external programs. Constructs CGI environment variables from the request, pipes the request body to the program's stdin, reads the program's stdout for the response (parsing CGI response headers and body), and handles the program's stderr for error logging. Enforces a configurable CGI script timeout (30 seconds default)
  - **`FizzBuzzAPIHandler`**: the primary request handler for the evaluation API. Routes `GET /api/v1/evaluate?n=<number>` to the `FizzBuzzServiceBuilder`-configured evaluation pipeline, returning JSON responses (`{"input": 15, "result": "FizzBuzz", "rules_applied": ["fizz", "buzz"]}`). Routes `GET /api/v1/evaluate/range?start=<n>&end=<m>` for batch evaluation. Routes `GET /api/v1/health` for health check responses. Routes `GET /api/v1/metrics` for Prometheus-compatible metrics exposition. All API responses include `X-FizzBuzz-Evaluation-Time`, `X-FizzBuzz-Cache-Hit`, and `X-FizzBuzz-Request-Id` headers
- **WebSocket Upgrade**: Bidirectional real-time communication:
  - **`WebSocketUpgradeHandler`**: detects WebSocket upgrade requests (`Connection: Upgrade`, `Upgrade: websocket`) and performs the RFC 6455 handshake: validates the `Sec-WebSocket-Key` header, computes the `Sec-WebSocket-Accept` response by SHA-1 hashing the key concatenated with the WebSocket GUID, and responds with `101 Switching Protocols`. After the handshake, the connection transitions from HTTP to WebSocket frame protocol
  - **`WebSocketFrameCodec`**: encodes and decodes WebSocket frames per RFC 6455: FIN bit, RSV bits, opcode (text=0x1, binary=0x2, close=0x8, ping=0x9, pong=0xA), masking (server-to-client frames are unmasked, client-to-server frames must be masked), and payload length (7-bit, 16-bit extended, 64-bit extended). Handles frame fragmentation for messages exceeding the maximum frame payload size (64KB default)
  - **`FizzBuzzStreamEndpoint`**: a WebSocket endpoint at `/ws/stream` that provides real-time FizzBuzz evaluation streaming. Clients send numbers (or ranges) as text frames and receive evaluation results as they complete. Supports subscription to the platform's event bus, allowing clients to observe evaluation events, cache state transitions, and system metrics in real time
- **Connection Management**: Persistent connections and pooling:
  - **`ConnectionPool`**: manages a pool of active TCP connections from FizzNet. Configurable parameters: `max_connections` (maximum concurrent connections, default 1024), `max_connections_per_host` (for upstream proxy connections, default 64), `idle_timeout` (time before an idle connection is closed, default 60 seconds), `max_keepalive_requests` (maximum requests per keep-alive connection, default 1000). The pool tracks connection state: `ACTIVE` (serving a request), `IDLE` (waiting for a new request on a keep-alive connection), `CLOSING` (sending remaining response data before close), `CLOSED`
  - **`KeepAliveManager`**: implements HTTP/1.1 persistent connections per RFC 7230 Section 6.3. Connections default to keep-alive unless the client sends `Connection: close`. After sending a response, the manager returns the connection to the idle pool for reuse. Idle connections are evicted after `idle_timeout` or when the pool reaches capacity. The `Keep-Alive` response header includes `timeout` and `max` parameters informing the client of server-side limits
  - **`HTTP2ConnectionManager`**: manages HTTP/2 connections with multiplexed streams. Each connection supports up to `max_concurrent_streams` (default 128) simultaneous request/response exchanges. Implements connection-level and stream-level flow control via `WINDOW_UPDATE` frames. Handles stream prioritization, dependency trees, and weight-based scheduling. Sends periodic `PING` frames for connection liveness detection
- **Transfer Encoding**: Chunked transfer and content encoding:
  - **`ChunkedTransferEncoder`**: encodes response bodies using HTTP/1.1 chunked transfer encoding for responses where the total body size is not known in advance (streaming responses, server-sent events, large file downloads generated on the fly). Each chunk is formatted as `chunk-size CRLF chunk-data CRLF`, terminated by a zero-length chunk and optional trailer headers
  - **`ContentEncoder`**: compresses response bodies based on the client's `Accept-Encoding` header. Supports three compression algorithms: `gzip` (RFC 1952), `deflate` (RFC 1951), and `br` (Brotli, RFC 7932). Selects the best algorithm by client preference and server-side priority (brotli > gzip > deflate). Sets `Content-Encoding` and adjusts `Content-Length` (or switches to chunked transfer if compressing a streaming response). Compression is skipped for responses smaller than 1KB (configurable via `--fizzweb-compression-min-size`) or for already-compressed content types (images, video, archives)
  - **`ContentNegotiator`**: implements HTTP content negotiation per RFC 7231 Section 5.3. Selects response content type based on `Accept` header quality values. For the FizzBuzz API, supports `application/json` (default), `text/plain`, `text/html` (rendered evaluation result), `application/xml`, and `text/csv` (for range evaluation results)
- **Access Logging**: Structured request/response logging:
  - **`AccessLogger`**: logs every request/response cycle in a structured format. Supports three log formats: `combined` (Apache Combined Log Format: `%h %l %u %t "%r" %>s %b "%{Referer}i" "%{User-Agent}i"`), `json` (structured JSON with all request/response fields, timing data, and trace context), and `fizzbuzz` (custom format including evaluation result and cache hit status). Logs are emitted to the platform's event sourcing journal via the event bus and optionally to a per-virtual-host log file. Each log entry includes: remote IP, timestamp, request method, URL, HTTP version, status code, response size, referrer, user agent, response time (microseconds), upstream response time (if proxied), TLS protocol version, and OpenTelemetry trace ID
  - **`AccessLogRotator`**: rotates access log files based on size (default 100MB) or time (daily rotation). Compressed rotated logs with gzip. Maintains configurable retention (default 30 days / 10 rotated files)
- **Rate Limiting Integration**: Traffic shaping at the server level:
  - **`ServerRateLimiter`**: integrates with the platform's existing rate limiting infrastructure to enforce request rate limits at the HTTP server level. Supports per-IP, per-virtual-host, and per-endpoint rate limiting. When a request exceeds the rate limit, responds with `429 Too Many Requests` and a `Retry-After` header indicating when the client may retry. Rate limit state is shared across connections via the platform's CRDT-based rate limiter for consistency in multi-instance deployments
- **Server Middleware Pipeline**: Composable request/response processing:
  - **`ServerMiddlewarePipeline`**: a chain of middleware components that process requests before handler dispatch and responses before serialization. Built-in middleware includes: `SecurityHeadersMiddleware` (adds `X-Frame-Options`, `X-Content-Type-Options`, `X-XSS-Protection`, `Content-Security-Policy`, `Referrer-Policy`), `CORSMiddleware` (handles preflight `OPTIONS` requests and sets `Access-Control-*` headers), `RequestIdMiddleware` (generates or propagates `X-Request-Id` for distributed tracing), `ETTagMiddleware` (generates `ETag` headers for cacheable responses and handles `If-None-Match` conditional requests). The pipeline integrates with the platform's existing `IMiddleware` interface
  - **`FizzWebMiddleware`** (IMiddleware implementation): integrates FizzWeb with the platform's main middleware pipeline. Records HTTP server metrics (requests per second, average response time, active connections, bytes transferred) in the processing context for downstream middleware and observers
- **Graceful Shutdown**: Zero-downtime server lifecycle:
  - **`GracefulShutdownManager`**: coordinates server shutdown without dropping in-flight requests. On receiving a shutdown signal (SIGTERM from FizzContainerd, or `--fizzweb-shutdown` CLI), the manager: (1) stops accepting new connections, (2) sets a `Connection: close` header on all in-progress responses to prevent new requests on keep-alive connections, (3) waits for all in-flight requests to complete (with a configurable timeout, default 30 seconds), (4) forcefully terminates remaining connections if the timeout expires, (5) closes the listening socket. Shutdown progress is reported to FizzContainerd's task service for container lifecycle coordination. Health check endpoints return `503 Service Unavailable` during the drain period to signal load balancers to stop routing traffic
  - **`ServerLifecycle`**: manages the complete server lifecycle: `STARTING` (binding ports, loading certificates, initializing connection pool), `RUNNING` (accepting and serving requests), `DRAINING` (graceful shutdown in progress), `STOPPED` (all resources released). Emits lifecycle events to the platform's event bus at each transition
- **Server Configuration**: Centralized server tuning:
  - **`FizzWebConfig`**: server configuration data model aggregating all tunable parameters: listen addresses and ports, TLS configuration, virtual host definitions, connection pool limits, timeout values (read timeout, write timeout, idle timeout, header timeout), compression settings, rate limiting profiles, logging configuration, and shutdown grace period. Configuration is loaded from the platform's `ConfigurationManager` with CLI flag overrides
- **CLI Flags**: `--fizzweb` (enable the HTTP server), `--fizzweb-port <port>` (HTTP listen port, default 8080), `--fizzweb-tls-port <port>` (HTTPS listen port, default 8443), `--fizzweb-host <addr>` (bind address, default 0.0.0.0), `--fizzweb-force-tls` (redirect HTTP to HTTPS), `--fizzweb-workers <n>` (number of worker connections, default 4), `--fizzweb-max-connections <n>` (maximum concurrent connections), `--fizzweb-keepalive-timeout <sec>` (keep-alive idle timeout), `--fizzweb-document-root <path>` (static file root directory), `--fizzweb-autoindex` (enable directory listing), `--fizzweb-compression-min-size <bytes>` (minimum response size for compression), `--fizzweb-access-log-format <combined|json|fizzbuzz>` (access log format), `--fizzweb-vhosts <config>` (virtual host configuration file), `--fizzweb-cgi-dir <path>` (CGI script directory), `--fizzweb-websocket` (enable WebSocket support), `--fizzweb-rate-limit <n>` (requests per second per IP), `--fizzweb-shutdown-timeout <sec>` (graceful shutdown timeout), `--fizzweb-cors-origins <origins>` (allowed CORS origins), `--fizzweb-h2` (enable HTTP/2 support)

### Why This Is Necessary

Because the Enterprise FizzBuzz Platform has built the entire web infrastructure stack except the web server. FizzNet provides reliable TCP delivery. FizzDNS resolves hostnames. FizzProxy load-balances across backend pools. The API gateway transforms and validates requests. FizzCNI configures container network interfaces with port mappings. The service mesh establishes sidecar proxies with mTLS, circuit breaking, and retry logic. The rate limiter enforces traffic quotas. FizzOTel instruments distributed traces. The secrets vault stores TLS certificates. All of these subsystems exist to support HTTP request serving. None of them can function without an HTTP server to receive and process those requests.

The absence of an HTTP server means the platform's external API surface exists only as an interface definition. The FizzBuzz-as-a-Service API endpoints defined in Round 1 cannot be reached by any external client. The OpenAPI specification generated by the platform describes routes that no server implements. The monitoring dashboard exists as HTML/CSS/JavaScript assets that no server can serve. The platform evaluates FizzBuzz through CLI invocation exclusively -- there is no network-accessible entry point for programmatic access.

Every production application framework -- Django, Flask, Express, Spring Boot, Rails, ASP.NET -- includes or integrates with an HTTP server. The HTTP server is not optional infrastructure. It is the fundamental interface between a networked application and its clients. A platform with 300,000+ lines of code, 116 infrastructure modules, a complete TCP/IP stack, a DNS server, a reverse proxy, a service mesh, a container runtime, and a deployment pipeline that cannot serve a single HTTP request represents an architectural gap of the first order. FizzWeb closes that gap. It is the server that all other network infrastructure components were built to support.
