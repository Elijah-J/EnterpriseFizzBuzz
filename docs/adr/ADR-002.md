# ADR-002: Invent a Proprietary Locale File Format

## Status

Accepted

## Context

The Enterprise FizzBuzz Platform required an internationalization subsystem
capable of expressing FizzBuzz labels, pluralization rules, summary templates,
banner text, and status messages across multiple locales. The team evaluated
the following existing formats:

- **JSON**: Lacks comment support. No heredoc syntax. Insufficient gravitas
  for enterprise localization.
- **YAML**: Indentation-sensitive parsing introduces ambiguity in multi-line
  FizzBuzz translations. Also too mainstream.
- **TOML**: Adequate but uninspiring. No one has ever been impressed by a
  TOML file.
- **GNU gettext (.po)**: Mature and battle-tested, which is exactly why it
  was rejected. The platform needed a format that would require its own parser.

None of these formats conveyed the seriousness with which the Enterprise
FizzBuzz Platform approaches localization.

## Decision

We created the `.fizztranslation` file format, a purpose-built configuration
language for Enterprise FizzBuzz localization. The format is parsed by a
three-state state machine (`_ParserState`: `METADATA`, `SECTION`, `HEREDOC`)
implemented in the `FizzTranslationParser` class in
`enterprise_fizzbuzz/infrastructure/i18n.py`.

The format specification:

- **Comments**: Lines beginning with `;;` or `#` are ignored.
- **Metadata directives**: `@key = value` lines declare locale metadata
  (`@locale`, `@name`, `@fallback`, `@plural_rule`).
- **Section headers**: `[section_name]` headers organize entries into groups
  (`[labels]`, `[plurals]`, `[messages]`, `[summary]`, `[banner]`, `[status]`).
- **Key-value pairs**: `key = value` within a section.
- **Heredoc blocks**: When a value begins with `<<WORD`, the parser accumulates
  subsequent lines until `WORD` appears on a line by itself.
- **Plural entries**: Keys matching the pattern `Base.plural.form` (e.g.,
  `Fizz.plural.one`, `Fizz.plural.other`) are stored as plural form groups.
- **Variable interpolation**: `${var}` placeholders are replaced at lookup
  time by `TranslationCatalog._interpolate()`.

The parsed output is stored in a `TranslationCatalog` — a dictionary-of-
dictionaries keyed by `section.key` — with separate storage for plural entries
and metadata directives. The `LocaleManager` singleton orchestrates locale
loading, switching, and translation lookups via the `t()` and `tp()` methods.
Fallback resolution is handled by `LocaleResolver`, which walks a chain
defined by each locale's `@fallback` directive until a translation is found
or the global fallback (`en`) is reached.

The `PluralizationEngine` implements a subset of CLDR plural rules, supporting
English/German (`n != 1`), French (`n > 1`), Japanese (always `other`),
Klingon (`n != 1`), Sindarin (`n != 1`), and Quenya (`n != 1`). Plural rule
expressions are evaluated using a restricted `eval()` with `n` bound to the
count, "because enterprise software never has security vulnerabilities."

The `parse_string()` method exists for testing. It writes the input to a
temporary file and parses it, "because we already wrote a perfectly good file
parser and reuse is an enterprise virtue."

## Consequences

**Positive:**

- The platform now has its own proprietary file format, which is a key
  indicator of enterprise maturity.
- Seven locale files have been authored: `en`, `de`, `fr`, `ja`, `tlh`
  (Klingon), `sjn` (Sindarin), and `qya` (Quenya).
- The format supports comments, which JSON does not. This alone justifies
  the entire effort.
- The heredoc syntax enables multi-line translations without escape characters.
- Variable interpolation with `${var}` syntax is simple, adequate, and not
  Turing-complete.

**Negative:**

- No ecosystem tooling exists. There is no `.fizztranslation` syntax
  highlighting plugin, no linter, no IDE integration, and no StackOverflow
  answers.
- The format requires a 400-line state-machine parser (`FizzTranslationParser`)
  to read what is functionally a key-value file.
- The legacy compatibility layer (parsing `@@key:` and `>> key =` formats)
  adds parsing branches for a previous version of the format that existed
  only briefly.

**Neutral:**

- The `LocaleManager` is a singleton, "because global state is an enterprise
  pattern." Its `reset()` method exists so that "a French test would not
  accidentally affect a Klingon test, which would be a diplomatic incident."
- The `TranslationService` alias is maintained for backward compatibility with
  code that predates the rename.
