# ADR-007: Write Cache Eulogies for Evicted Entries

## Status

Accepted

## Context

The Enterprise FizzBuzz Platform's caching layer
(`enterprise_fizzbuzz/infrastructure/cache.py`) implements multiple eviction
policies for managing cache capacity: LRU, LFU, FIFO, and DramaticRandom.
When the cache reaches capacity and a new entry must be stored, an existing
entry is selected for eviction by the active policy.

In most caching systems, eviction is a silent operation. The entry is removed,
its memory reclaimed, and the system continues without ceremony. The
architecture review board determined that this approach was insufficient.
No data should be garbage-collected without a proper farewell. The evicted
entry served the platform faithfully — it cached the result of a modulo
operation so that the modulo operation would not need to be performed again —
and its departure should be acknowledged with the solemnity it deserves.

## Decision

We implemented the `EulogyGenerator` class in
`enterprise_fizzbuzz/infrastructure/cache.py`. When a cache entry is evicted,
the generator composes a satirical obituary using the entry's life statistics.

The `EulogyGenerator.compose()` class method accepts a `CacheEntry` and the
name of the eviction policy responsible for its demise. It selects from a pool
of 10 eulogy templates, each a unique narrative form:

1. A straightforward obituary noting faithful service and cause of death
2. A "loving memory" format tracking dignity at time of departure
3. An appeal to the garbage collector for respectful treatment
4. A "moment of silence" format noting the entry's dignified birth and
   undignified end
5. An obituary in news-wire style ("OBITUARY: Cache entry...")
6. A reflection on the cache life that was not chosen but endured
7. A meditation on the dream of eternal caching, cut short
8. A breaking-news format reporting the forced eviction
9. A theatrical farewell referencing "this mortal hash map"
10. A formal eviction notice from cache management

Each template interpolates the following variables from the `CacheEntry`:

| Variable        | Source                          | Description                                   |
|-----------------|---------------------------------|-----------------------------------------------|
| `{key}`         | `entry.key`                     | The cache key                                 |
| `{output}`      | `entry.result.output`           | The cached FizzBuzz output (e.g., "FizzBuzz") |
| `{number}`      | `entry.result.number`           | The number that was evaluated                 |
| `{access_count}`| `entry.access_count`            | Times the entry was accessed                  |
| `{age}`         | `entry.get_age_seconds()`       | Lifetime in seconds                           |
| `{dignity}`     | `entry.dignity_level`           | Dignity remaining at eviction (0.0-1.0)       |
| `{policy}`      | `policy_name` parameter         | The eviction policy that selected this victim |

Before composing the eulogy, `entry.update_dignity()` is called to recalculate
the dignity level based on the entry's age relative to its TTL. Dignity degrades
linearly: an entry at half its TTL has 50% dignity remaining. An expired entry
has no dignity whatsoever.

If eulogy composition fails for any reason (template formatting error, missing
attribute), a `CacheEulogyCompositionError` is raised. The existence of a
dedicated exception class for eulogy composition failures reflects the
platform's commitment to treating every subsystem, no matter how absurd, as
a first-class failure domain.

The `CacheEntry` dataclass includes the following eulogy-relevant fields:

- `dignity_level: float = 1.0` — Starts at full dignity, degrades with age
- `eulogy: Optional[str] = None` — Populated at eviction time
- `ttl_seconds: float = 3600.0` — Default one-hour TTL, used for dignity
  calculation

The `DramaticRandomPolicy` eviction strategy is specifically designed to pair
with the eulogy system: it selects a random victim and logs a WARNING-level
eulogy, adding "an element of suspense to every cache operation."

## Consequences

**Positive:**

- Evicted cache entries are mourned. Their service is acknowledged. Their
  dignity levels are tracked and reported at time of departure.
- The eulogy pool of 10 templates provides sufficient variety that consecutive
  evictions produce distinct farewell messages, reducing eulogy fatigue among
  log readers.
- The dignity tracking system provides a novel cache health metric: average
  dignity at eviction. Low average dignity indicates entries are being evicted
  near or past their TTL, which is healthy. High average dignity indicates
  entries are being evicted in their prime, which may indicate insufficient
  cache capacity.
- Template #4 ("Rest in /dev/null") and template #9 ("shuffled off this mortal
  hash map") demonstrate that the eulogy system takes its literary obligations
  seriously.

**Negative:**

- Each eviction now requires string formatting, random template selection,
  dignity recalculation, and log emission, adding non-trivial overhead to an
  operation that frees space for caching the result of `n % 3`.
- The eulogy templates are hardcoded. There is no `.fizzeulogy` file format
  for customizing eviction eulogies, though the precedent set by ADR-002
  suggests this is only a matter of time.
- Log files in high-eviction scenarios will contain more eulogy text than
  FizzBuzz evaluation output, potentially overwhelming log aggregation
  systems that were not designed for literary analysis.

**Neutral:**

- Disabling eulogies is technically possible by subclassing `EvictionPolicy`
  and overriding the eviction path to skip eulogy composition. The
  `EulogyGenerator` docstring describes this as "technically possible but
  ethically questionable."
- The `CacheEulogyCompositionError` exception is one of 69 custom exception
  classes in the platform. Its error code and integration with the exception
  hierarchy is documented in the exception catalog.
