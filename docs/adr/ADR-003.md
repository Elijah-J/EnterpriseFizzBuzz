# ADR-003: Implement MESI Cache Coherence for a Single-Process Cache

## Status

Accepted

## Context

The Enterprise FizzBuzz Platform's caching layer stores the results of FizzBuzz
evaluations in an in-memory dictionary. The cache operates within a single
Python process. There is one cache. There are zero concurrent caches. There are
no other processors, cores, or nodes that need to observe or invalidate cache
entries.

The caching layer nonetheless required a coherence protocol. Without one, there
would be no state machine to diagram, no transition events to publish, and no
coherence violations to throw exceptions about. The architecture review board
determined that an uncoordinated in-memory dictionary was insufficient for
enterprise use, citing "aspirational multi-cache readiness" as justification.

## Decision

We implemented the full Modified/Exclusive/Shared/Invalid (MESI) cache
coherence protocol in `enterprise_fizzbuzz/infrastructure/cache.py` via the
`CacheCoherenceProtocol` class. The four states are defined in the
`CacheCoherenceState` enum in `enterprise_fizzbuzz/domain/models.py`.

Each `CacheEntry` carries a `coherence_state` field (defaulting to
`CacheCoherenceState.EXCLUSIVE`) alongside its cached `FizzBuzzResult`, access
statistics, a `dignity_level` float in `[0.0, 1.0]` that degrades linearly
over the TTL period, and an optional eulogy string.

The valid state transitions are:

| From       | To         | Trigger                                          |
|------------|------------|--------------------------------------------------|
| INVALID    | EXCLUSIVE  | Cache miss ŌĆö entry loaded from the modulo oracle  |
| EXCLUSIVE  | MODIFIED   | Entry updated                                    |
| EXCLUSIVE  | SHARED     | Another reader appears (hypothetically)           |
| EXCLUSIVE  | INVALID    | Entry invalidated                                |
| SHARED     | MODIFIED   | Entry updated, other sharers invalidated          |
| SHARED     | INVALID    | Entry invalidated                                |
| MODIFIED   | EXCLUSIVE  | Write-back completed                             |
| MODIFIED   | SHARED     | Write-back + share                               |
| MODIFIED   | INVALID    | Entry invalidated                                |

Invalid transitions raise `CacheCoherenceViolationError`. All transitions are
logged at DEBUG level and, if an `IEventBus` is provided, published as
`EventType.CACHE_COHERENCE_TRANSITION` events with a payload containing the
entry key, old state, new state, and transition sequence number.

The `CacheStore` class enforces coherence state on every operation:

- **`put()`**: New entries enter as `EXCLUSIVE`. Existing entries are
  transitioned to `MODIFIED` before update, then reset to `EXCLUSIVE`.
- **`invalidate()`**: Entries are transitioned to `INVALID` before removal.
  If the transition fails (entry already `INVALID`), the entry's state is
  force-set to `INVALID` and removal proceeds.

The `get_state_distribution()` method returns a count of entries in each MESI
state, suitable for dashboard rendering.

The cache also supports four eviction policies via the Strategy pattern:

- `LRUPolicy` ŌĆö Least Recently Used
- `LFUPolicy` ŌĆö Least Frequently Used
- `FIFOPolicy` ŌĆö First-In-First-Out
- `DramaticRandomPolicy` ŌĆö "The Hunger Games"

Policies are instantiated by `EvictionPolicyFactory.create()`, which maps
string names (`lru`, `lfu`, `fifo`, `dramatic_random`) to policy classes.

Thread safety is provided by fine-grained locking. The coherence protocol
maintains its own `threading.Lock` and tracks a global `transition_count`.

## Consequences

**Positive:**

- The caching layer achieves full MESI protocol compliance, as verified by
  the valid transition table in `CacheCoherenceProtocol._VALID_TRANSITIONS`.
- Cache coherence transitions are observable through the event bus, enabling
  downstream monitoring dashboards.
- The state distribution method provides real-time visibility into the
  coherence posture of the cache fleet (population: 1).

**Negative:**

- The SHARED state is never reached in production. No second cache exists to
  share entries with. The transition from EXCLUSIVE to SHARED is triggered only
  when "another reader appears, hypothetically."
- The coherence protocol adds locking overhead and event publication cost to
  every cache put and invalidate operation, for a cache whose contents could
  be recomputed in the time it takes to acquire the lock.
- The `CacheCoherenceViolationError` exception has been defined, imported, and
  documented, but raising it in a single-cache system requires an internal
  logic error ŌĆö which means the exception exists primarily to catch bugs in
  the coherence protocol that would not exist if the coherence protocol did
  not exist.

**Neutral:**

- Intel's L1 cache uses MESI. The Enterprise FizzBuzz Platform uses MESI.
  The comparison ends there, but it can be made.
