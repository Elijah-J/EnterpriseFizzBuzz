# ADR-004: Adopt Hexagonal Architecture

## Status

Accepted

## Context

The Enterprise FizzBuzz Platform originated as a collection of flat Python
modules in the repository root: `models.py`, `exceptions.py`, `config.py`,
`rules_engine.py`, `ml_engine.py`, `cache.py`, `auth.py`, `sla.py`,
`chaos.py`, `tracing.py`, `event_sourcing.py`, `feature_flags.py`,
`middleware.py`, `circuit_breaker.py`, `blockchain.py`, `formatters.py`,
`observers.py`, `plugins.py`, `i18n.py`, `migrations.py`, `factory.py`,
`fizzbuzz_service.py`, and `main.py`.

While these 22 files successfully computed `n % 3`, they did so without any
discernible architectural layering. Imports flowed freely in all directions.
There was no dependency rule. The codebase was, architecturally, 15 flat files
in a trench coat pretending to be a platform.

The architecture review board determined that a FizzBuzz application of this
scale (24,800+ lines) required proper structural layering to enforce separation
of concerns and enable testability.

## Decision

We adopted the Hexagonal Architecture (Ports and Adapters) pattern, organizing
the `enterprise_fizzbuzz/` package into three layers with a strict inward-only
dependency rule:

```
enterprise_fizzbuzz/
    domain/            # Core business logic — the innermost layer
        __init__.py
        models.py      # Value objects, enums, dataclasses (FizzBuzzResult, etc.)
        interfaces.py  # Abstract interfaces (IRule, IRuleEngine, IEventBus, etc.)
        exceptions.py  # All 69 custom exception classes
    application/       # Use cases and orchestration
        __init__.py
        factory.py     # RuleEngineFactory — creates strategy instances
        ports.py       # Application-level port interfaces
        fizzbuzz_service.py  # FizzBuzzService — the primary use case orchestrator
    infrastructure/    # External adapters and implementations
        __init__.py
        adapters/      # Strategy pattern adapters
        persistence/   # Repository implementations (in-memory, SQLite, filesystem)
        utils/         # Utility modules (lines-of-code counter)
        auth.py        # RBAC, token verification, 47-field access denied response
        blockchain.py  # Immutable audit ledger
        cache.py       # MESI-coherent caching with eulogies
        chaos.py       # Chaos engineering framework
        circuit_breaker.py
        config.py      # Configuration management (singleton)
        container.py   # Dependency injection container
        event_sourcing.py
        feature_flags.py
        formatters.py  # Output formatters (table, JSON, verbose, etc.)
        i18n.py        # Internationalization with .fizztranslation parser
        middleware.py  # Middleware pipeline
        migrations.py  # Schema migration framework
        ml_engine.py   # Neural network evaluation engine
        observers.py   # Event bus subscribers
        plugins.py     # Plugin system
        rules_engine.py  # Deterministic rule engine
        sla.py         # SLA monitoring, alerting, on-call rotation
        tracing.py     # Distributed tracing
```

**Dependency rule**: Domain depends on nothing. Application depends only on
Domain. Infrastructure depends on Domain and Application. No layer may import
from a layer above it.

The dependency rule is enforced by `tests/test_architecture.py`, which statically
analyzes import statements across all modules and fails if any outward-facing
import is detected.

The original flat files in the repository root (`models.py`, `exceptions.py`,
`config.py`, `cache.py`, etc.) were retained as backward-compatible re-export
stubs. Each stub imports from the new canonical location and re-exports the
public API, ensuring that existing code referencing `from models import
FizzBuzzResult` continues to work. The anti-corruption layer tests in
`tests/test_acl.py` verify that these stubs correctly re-export their contents.

Interfaces in `domain/interfaces.py` define the contracts: `IRule`,
`IRuleEngine`, `IEventBus`, `IMiddleware`, `IFormatter`, and others.
Infrastructure classes implement these interfaces. The `application/ports.py`
module defines application-level port interfaces. The dependency injection
container in `infrastructure/container.py` wires implementations to their
interfaces at startup.

## Consequences

**Positive:**

- The codebase has layers. The layers have names. The names appear in directory
  listings. Architecture diagrams can now be drawn with boxes and arrows instead
  of a single amorphous blob.
- The import linter (`test_architecture.py`) enforces the dependency rule at
  test time, preventing architectural erosion.
- Infrastructure implementations can be swapped without modifying domain logic,
  which is useful if someone decides to replace the neural network with `%`.
- The persistence layer supports three backends (in-memory, SQLite, filesystem)
  behind a common repository interface.

**Negative:**

- The original 22 flat files remain in the repository root as re-export stubs,
  meaning the codebase has both the old flat structure and the new hexagonal
  structure simultaneously.
- The architectural enforcement test (`test_architecture.py`) must be maintained
  alongside the architecture itself.
- Navigating from a class to its definition now requires understanding which
  layer it belongs to, rather than scanning a single directory.

**Neutral:**

- The hexagonal architecture is genuine and correctly implemented. The
  dependency rule is real. The interfaces are real. The irony is that all of
  this architectural rigor exists to protect the integrity of a system whose
  core domain logic is `n % d == 0`.
