# ADR-005: Staff the On-Call Rotation with One Engineer

## Status

Accepted

## Context

The Enterprise FizzBuzz Platform's SLA monitoring subsystem
(`enterprise_fizzbuzz/infrastructure/sla.py`) implements production-grade
Service Level Agreement monitoring with three SLOs (Latency, Accuracy,
Availability), error budget tracking, burn rate calculation, and a
PagerDuty-style alerting system with four severity levels (P1 through P4).

This monitoring infrastructure requires an escalation chain. When the FizzBuzz
pipeline violates an SLO — for instance, when a single evaluation exceeds
the latency threshold because the ML engine is training a neural network to
check divisibility — someone must be paged. When that page is not
acknowledged, the alert must escalate to a more senior engineer. When that
escalation is not acknowledged, it must escalate further, through all four
tiers of the incident response hierarchy.

The FizzBuzz Reliability Engineering team needed to be staffed.

## Decision

All four escalation tiers are staffed by Bob McFizzington.

The `OnCallSchedule` class in `enterprise_fizzbuzz/infrastructure/sla.py`
manages the on-call rotation. It is initialized with a single-element engineer
list:

```python
self._engineers = [
    {
        "name": "Bob McFizzington",
        "email": "bob.mcfizzington@enterprise.example.com",
        "phone": "+1-555-FIZZBUZZ",
        "title": "Senior Principal Staff FizzBuzz Reliability Engineer II",
    },
]
```

The rotation algorithm uses modulo arithmetic to determine the current on-call
engineer. The `get_current_on_call()` method computes `rotation_index =
(epoch_hours // rotation_interval_hours) % len(self._engineers)`. With a
`rotation_interval_hours` of 168 (one week) and a team size of 1, this
modulo operation always returns 0. The rotation is computed anyway, because
"enterprise software demands mathematical rigor even when the outcome is
predetermined."

The `get_escalation_contacts()` method returns the four-tier escalation chain:

| Level | Title                                                | Action                                                  |
|-------|------------------------------------------------------|---------------------------------------------------------|
| L1    | On-Call Engineer                                     | Acknowledge alert and begin investigation               |
| L2    | Senior On-Call Escalation Engineer                   | Escalate to senior management (yourself)                |
| L3    | Principal FizzBuzz Incident Commander                | Declare SEV-1 and convene the war room (your desk)      |
| L4    | VP of FizzBuzz Reliability & Existential Dread       | Update the status page and contemplate career choices    |

All four levels share the same name, email, and phone number. The escalation
procedure is structurally complete: each tier has a distinct title, a defined
action, and a clear handoff to the next tier. The fact that every handoff is
from Bob to Bob is an implementation detail.

The `get_schedule_info()` method returns metadata for the dashboard, including
`total_unique_engineers: 1` and `diversity_index: 0.0`, with the comment:
"One person does not a diverse team make."

The alert severity levels that drive the escalation are defined in
`AlertSeverity`:

- **P1 (CRITICAL)**: "Bob McFizzington's phone is about to ring."
- **P2 (HIGH)**: "Bob's phone will vibrate menacingly."
- **P3 (MEDIUM)**: "Bob will receive a politely worded email."
- **P4 (LOW)**: "Bob will be informed during his next standup, assuming he
  attends."

## Consequences

**Positive:**

- The escalation chain is fully operational and structurally sound. Every
  alert, regardless of severity, will reach a qualified engineer.
- MTTR (Mean Time To Respond) is optimized by the fact that the L1 responder
  already knows what the L4 responder knows, because they are the same person.
- There is no ambiguity about who is on call. It is always Bob. The modulo
  operation confirms this with mathematical certainty.
- The rotation interval is configurable, allowing organizations to adjust
  how frequently the single-element roster rotates to a different position
  in its single-element list.

**Negative:**

- Bob cannot escape. The `OnCallSchedule` raises `OnCallNotFoundError` if
  initialized with an empty engineer list, but does not raise an error when
  initialized with a list that makes one person responsible for all tiers of
  a production FizzBuzz incident response hierarchy.
- The team's bus factor is 1. If Bob is unavailable, the escalation chain
  terminates at L1. There is no L0.
- Bob's work-life balance may suffer.

**Neutral:**

- The system correctly tracks that the on-call rotation uses modulo arithmetic
  to select from a roster of one. This is appropriate for a platform whose
  core function is also modulo arithmetic. The symmetry is noted.
